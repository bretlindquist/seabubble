# Surgical Plan: Chunk 3 - XNU Audit Token & Identity Validation

**Date:** April 26, 2026
**Target:** `~/git/seabubble/daemon/security-islandd`
**Objective:** Replace the stubbed authentication in `security-islandd` with cryptographically binding OS-level checks. The daemon must extract the `audit_token_t` from the incoming socket connection, mathematically prove the PID, and validate the `session_nonce` generated by the launcher.

## 1. Agent Assignments
- **Agent 1 (Systems/Rust):**
  - Update `security-islandd/src/main.rs`.
  - Implement a `ConnectionIdentity` struct that stores `peer_pid`, `peer_uid`, and `session_nonce_validated`.
  - Use `getsockopt` with `LOCAL_PEERTOKEN` (via `libc::audit_token_t`) to retrieve the OS-level token.
  - Extract the exact `pid` from the audit token (using `audit_token_to_pid` via `libc`).
  - Require the first JSON payload over the socket to be a `security.identify` handshake containing the nonce.
  - Drop connections immediately if the nonce is invalid or the PID cannot be proven.

- **Reviewer Agent:**
  - Verify memory safety (no raw pointer leaks when dealing with `libc`).
  - Ensure the connection is dropped gracefully without panicking the Tokio runtime.
  - Verify `cargo clippy -- -D warnings` passes.

## 2. Architecture Details
- Using `getpeereid()` is insufficient for preventing PID-reuse race conditions. By capturing the `audit_token_t` at accept time, we freeze the credentials of the process at that exact microsecond.
- The `ConnectionIdentity` struct becomes the immutable passport for that specific socket stream. We never query the OS for identity again after the handshake.
